### 1. Empty `NSPrivacyCollectedDataTypes` despite sending location off-device

**Files:** `HeadacheLogger/PrivacyInfo.xcprivacy`, `HeadacheLoggerWidget/PrivacyInfo.xcprivacy`

Both privacy manifests declare an empty `NSPrivacyCollectedDataTypes` array, but the app sends
approximate latitude/longitude to three Open-Meteo endpoints on every log and background forecast check:

- `https://api.open-meteo.com/v1/forecast`
- `https://archive-api.open-meteo.com/v1/archive`
- `https://air-quality-api.open-meteo.com/v1/air-quality`

Apple's guidelines require declaring any data type collected and sent off-device. `Precise Location`
must be listed with purpose `App Functionality`, linked to the Open-Meteo domain.

**Fix â€” add to both PrivacyInfo.xcprivacy files:**
```xml
<key>NSPrivacyCollectedDataTypes</key>
<array>
    <dict>
        <key>NSPrivacyCollectedDataType</key>
        <string>NSPrivacyCollectedDataTypePreciseLocation</string>
        <key>NSPrivacyCollectedDataTypeLinked</key>
        <false/>
        <key>NSPrivacyCollectedDataTypeTracking</key>
        <false/>
        <key>NSPrivacyCollectedDataTypePurposes</key>
        <array>
            <string>NSPrivacyCollectedDataTypePurposeAppFunctionality</string>
        </array>
    </dict>
</array>
```

### 2. Empty `NSPrivacyAccessedAPITypes` â€” UserDefaults used without declared reason

**Files:** `HeadacheLogger/PrivacyInfo.xcprivacy`, `HeadacheLoggerWidget/PrivacyInfo.xcprivacy`

`UserDefaults` is accessed extensively across every target via `HeadacheAppGroup.userDefaults`
and `@AppStorage` wrappers. Apple now requires every use of `UserDefaults` to be declared in
`NSPrivacyAccessedAPITypes` with a reason code. The appropriate reason code for app-group
shared defaults is `CA92.1` (accessing data from the same app's other targets).

**Fix â€” add to both PrivacyInfo.xcprivacy files:**
```xml
<key>NSPrivacyAccessedAPITypes</key>
<array>
    <dict>
        <key>NSPrivacyAccessedAPIType</key>
        <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
        <key>NSPrivacyAccessedAPITypeReasons</key>
        <array>
            <string>CA92.1</string>
        </array>
    </dict>
</array>
```

### 3. Watch app has no `PrivacyInfo.xcprivacy` at all

**Missing file:** `HeadacheLoggerWatch/PrivacyInfo.xcprivacy`

The `HeadacheLoggerWatch` target ships with zero privacy manifest. Apple has required a
privacy manifest for every executable in a submission bundle since Spring 2024. The watch
app uses `UserDefaults` (via shared app group) and `WatchConnectivity`, so it needs the
same manifest structure as the main app.

**Fix:** Create the file, then add it to `project.yml` under `HeadacheLoggerWatch` â†’ `sources`
as a resource (matching how the widget target includes it at line 72).

### 4. Paywall doesn't disclose trial-to-paid conversion

**File:** `HeadacheLogger/Views/PaywallView.swift`, `subscriptionDisclosure` computed property (~line 186)

The subscription disclosure text is static: *"Payment is charged to your Apple ID at confirmation.
The subscription auto-renews yearly at the listed price unless cancelled..."*

When a free trial is offered (detected via `hasFreeTrial(_:)`), Apple's guidelines explicitly
require stating that the trial will auto-convert to a paid subscription. The current text
mentions auto-renewal but doesn't connect it to the trial.

**Fix:** When `hasFreeTrial` is true, prepend or append something like:
*"After the 7-day free trial, this subscription automatically renews as a paid subscription."*

### 6. Widget Info.plist missing `ITSAppUsesNonExemptEncryption`

**File:** `HeadacheLoggerWidget/Info.plist`

Every executable in a submission must declare `ITSAppUsesNonExemptEncryption`. The main app
declares `<false/>`, but the widget extension doesn't. Apple can flag this during processing
and require the export compliance questionnaire.

**Fix:** Add `<key>ITSAppUsesNonExemptEncryption</key><false/>` to the widget's Info.plist.

### 7. Watch `CFBundleDisplayName` is just "Headache"

**File:** `HeadacheLoggerWatch/Info.plist`

The watch app's display name shows as "Headache" on the watch face and in notifications. The
main app is "One Tap Headache Tracker" â€” this stark single word looks out of place.

**Fix:** Change `CFBundleDisplayName` to "Headache Logger" to match the widget, or
"One Tap Headache" for brand consistency. The `CFBundleName` is `HeadacheLogger` which is
fine as the internal name.

### 8. Unnecessary `processing` background mode in Info.plist

**File:** `HeadacheLogger/Info.plist`

`UIBackgroundModes` declares both `fetch` and `processing`. The code only registers
`BGAppRefreshTask` (which needs `fetch`), not `BGProcessingTask` (which needs `processing`).
Having a declared-but-unused background capability can trigger reviewer questions about
what background processing is being done.

**Fix:** Remove `<string>processing</string>` from the `UIBackgroundModes` array.
